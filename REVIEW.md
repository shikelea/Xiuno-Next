# 代码审查报告

**日期:** 2026-03-04
**审查人:** Trae AI

## 执行摘要

该代码库是 Xiuno BBS 4.0 的现代化版本，旨在兼容 PHP 8.0+。在 Docker 化、Composer 集成和 API 开发方面已取得显著进展。然而，主要在数据库交互和密码哈希方面仍保留了一些旧有的安全实践。

## 主要发现

### 1. 安全性

- **SQL 注入风险**: 数据库抽象层 (`xiunophp/db.func.php`, `xiunophp/db_pdo_mysql.class.php`) 依赖 `addslashes()` 转义字符串值并手动构建 SQL 字符串。它**没有**使用 PDO 预处理语句进行变量插值。
    - *风险等级*: 高
    - *建议*: 重构 `db_` 函数以使用 PDO 预处理语句（`prepare()` 和 `execute()`），从根本上防止 SQL 注入。
- **弱密码哈希**: 用户密码使用带盐的 MD5 哈希 (`md5($password.$salt)`)。MD5 在密码学上被认为已损坏，容易受到碰撞和彩虹表攻击。
    - *风险等级*: 严重
    - *建议*: 迁移到 `password_hash()`（使用 Bcrypt 或 Argon2）和 `password_verify()`。这将需要制定针对现有用户的迁移策略（例如，在登录时重新哈希）。
- **会话安全**: 会话管理依赖于标准的 PHP 会话。确保在生产环境的 `php.ini` 中或通过 `ini_set()` 启用了 `session.cookie_httponly` 和 `session.cookie_secure`。

### 2. 架构与代码质量

- **现代化**: 向 PHP 8.0+ 的迁移显而易见且值得称赞。`composer.json` 和 `docker-compose.yml` 的包含极大地改善了开发体验。
- **API 设计**: 新的 `route/api/` 端点为无头交互提供了良好的基础。
- **全局状态**: 应用程序严重依赖全局变量和函数（`param()`, `db_*()`），这在传统 PHP 应用程序中很典型，但使得单元测试变得困难。

### 3. 可维护性

- **数据库驱动**: `db_pdo_mysql` 类设置 `sql_mode=''`（空），禁用了严格的 SQL 模式。这可能会掩盖数据截断或无效日期问题。
    - *建议*: 在开发环境中启用严格模式 (`STRICT_TRANS_TABLES`) 以尽早发现数据问题。

## 路线图建议

1.  **安全加固**: 优先升级密码哈希机制。
2.  **数据库层重构**: 逐步引入现代查询构建器或 ORM，或重构现有层以使用预处理语句。
3.  **测试**: 为关键路径引入自动化测试 (PHPUnit)，特别是新的 API 端点和工具函数。
